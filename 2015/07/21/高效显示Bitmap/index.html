Android-Bitmap的高效加载与处理

发表于 2015-07-21   |  

一、高效加载大图

图形有不同的形状与大小。在大多数情况下它们的实际大小都比需要呈现的尺寸大很多。系统的图库应用显示那些我们使用相机拍摄的照片，但是那些图片的分辨率通常都比设备屏幕的分辨率要高很多。
考虑到应用是在有限的内存下工作，我们只需要在内存中加载一个低分辨率的照片即可。为了更便于显示，这个分辨率的照片应该是与其对应的UI控件大小相适配。加载超过屏幕分辨率的高分辨率照片不仅对显示没有任何的提升，还会占用内存资源，此外在快速滑动图片时容易产生额外的效率问题。
那么，如何加载缩小版本的图片来避免超出程序的内存限制

（1）读取位图的尺寸和类型

为了避免java.lang.OutOfMemory的异常，我们需要在真正解析图片的之前获取图片的尺寸
BitmapFactory提供了一些解码（decode）的方法（decodeByteArray(),decodeFile(),decodeResource()）等，用来从不同的资源中创建一个Bitmap。我们应该根据图片的数据源来选择合适的解码方法。这些方法在构造位图的时候会尝试分配内存，因此容易导致OOM。每一种解码方法会同辉BimapFactory.Options设置一些附加的标记，以此来指定解码选项。设置inJustDecodeBounds属性为ture可以在解码的时候避免内存的分配，它会返回一个null的Bitmap，但是可以获取到outWidth，outHeight与outMimeType属性。你可以在构造Bitmap之前优先读取图片的尺寸和类型。

BitmapFactory.Options options = new BitmapFactory.Options();
options.inJustDecodeBounds = true;
BitmapFactory.decodeResource(getResource(),R.id.myimage,options);
int imageWidth = options.outWidth;
int imageHeight = options.outHeight;
String imageType = options.outMimeType;

（2）加载一个按比例缩小的版本到内存中

通过以上的步骤我们获取到了图片的尺寸和类型等参数，这些数据决定了我们是完整地将图片加载到内存中还是加载按比例缩小后的图片。在加载图片的时候我们要考虑到以下几点:

1.评估加载完整图片所消耗的内存。
2.程序在加载图片的时候可能涉及到其他的内存需求。
3.加载图片的控件的大小。
4.屏幕大小与当前设备的屏幕密度。

比如在使用128x96像素的ImageView加载像素1024x768的图片的时候，就没有必要加载完整的图片到内存中。
为了告诉解码器去加载一个缩小版本的图片到内存中，需要在BitmapFactory.Options中设置inSampleSize的值。例如，一个分辨率为2048x1536的图片，如果设置inSampleSize为4，那么会生成一个大约512x384大小的Bitmap。加载缩小后的图片消耗0.75MB的内存。如果加载原有的图片，则会消耗12MB左右的内存（假设Bitmap的配置为ARGB_8888)。

public static int calculateInSampleSize(
            BitmapFactory.Options options , int reqWidth, int reqHeight){
    // 获取到的图片的宽度和高度
    final int height = options.outHeight;
    final int width = options.outWidth;
    // 缩放比例
    int inSampleSize = 1;
    // 和加载图片的控件大小比较
    if(height > reqHeight || width > reqWidth){
        final int halfHeight = height / 2;
        final int halfWidth = width / 2;
        while((height / inSampleSize ) > reqHeight
                && (width / inSampleSize) > reqWidth){
            inSampleSize *= 2;
        }
    }
    return inSampleSize;
}

为了使用该方法，首先需要设置inJustDecodeBounds为true，获取Options的值。然后计算inSampleSize的值并设置inJustDecodeBounds为false。最后调用相关的解码方法。

public static Bitmap decodeSampledBitmapFromResource(Resource res , 
        int resId , int reqWidth , int reqHeight){
    final BitmapFactory.Options options = new BitmapFactory.Options();
    options.inJustDecodeBounds = true;
    BitmapFactory.decodeResource(res , resId , options);
    // 计算缩放比例
    options.inSampleSize = calculateInSampleSize(options , reqWidth , reqHeight);
    //
    options.inJustDecodeBounds = false;
    return BitmapFactory.decodeResource(res , resId , options);
}

使用上面可以简单地加载任意一张任意大小的图片。

//  显示100x100像素的缩略图
mImageView.setImageBitmap(decodeSampleBitmapFromResource(getResource(), R.id.myImage , 100, 100));

同样可以通过BitmapFactory.decode*方法来解析其他数据源的Bitmap

二、非UI线程处理Bitmap

加载网络或者存储卡时（或者是任何不在内存中的形式）都不应该在UI线程中执行。因为在上面种情况去加载数据时候，执行时间是不可估计的，依赖于许多因素（从网络或存储卡读取数据的速度，图片的大小，CPU的速度等）。如果其中任何一个子线程阻塞了UI线程，系统都会容易出现应用无响应的错误。下面介绍如果使用AsyncTask在后台线程中处理Bitmap并且演示如何处理并发的问题。

(1) 使用AysncTask(Use a AsyncTask)

AsyncTask类提供了一个在后台线程执行一些操作的简单方法，可以把后台的执行结果呈现到UI线程中。参考下面的例子来学习使用AsyncTask

class BitmapWorkerTask extends AsyncTask{
    private final WeakReference imageViewReference;
    private int data = 0 ;
    public BitmapWorkerTask(ImageView imageView){
        // 使用弱引用来保证ImageView可以被GC回收
        imageViewReference = new WeakReference(imageView);
    }
    // 后台解码图片
    @Override
    protected Bitmap doInBackground(Integer...params){
        data = params[0];
        return decodeSampledBitmapFromResource(getResource(),data , 100 ,100);
    }
    @Override
    protected void onPostExecute(Bitmap bitmap){
        if(imageViewReference != null && bitmap != null){
            final ImageView imageView = imageViewReference.get();
            if(imageView != null){
                imageView.setImageBitmap(bitmap);
            }
        }
    }
}

为ImageView使用WeakReference确保了AsyncTask所引用的资源可以被GC回收。因为任务结束时不能确保ImageView仍然存在，需要在onPostExecute()离检查引用是否存在。
异步加载Bitmap，只需要创建一个新的任务执行

public void loadBitmap(int resId,ImageView imageView){
    BitmapWorkerTask task = new BitmapWorkerTask(imageView);
    task.execute(resId);
}

(2) 处理并发问题

像类似ListView与GridView等视图控件在使用AsyncTask方法时，会带来并发的问题。在使用ListView和GridView时候item需要在滑动屏幕的时候重用。
解决并发问题的一种解决方法：ImageView保存最近使用的AsyncTask的引用。
创建一个专用的Drawable的子类来存储任务的引用。如下：使用BitmapDrawable,在任务执行的过程中，一个占位图会显示在ImageView中：

static class AsyncDrawable extends BitmapDrawable{
    private final WeakReference bitmapWorkerTaskReference;

    public AsyncDrawable(Resources res , Bitmap bitmap
            BitmapWorkerTask bitmapWorkerTask){
        super(res , bitmap);
        bitmapWorkerTaskReference = new WeakReference(bitmapWorkerTask);
    }

    public BitmapWorkerTask getBitmapWorkerTask(){
        return bitmapWorkerTaskReference.get();
    }
}

在执行BitmapWorkerTask之前，你需要创建一个AsyncDrawable并且将它绑定到目标控件ImageView中：

public void loadBitmap(int resId, ImageView imageView){
    if(cancelPotentialWork(resId, imageView)){
        final BitmapWorkerTask task = new BitmapWorkerTask(imageView);
        final AsyncDrawable asyncDrawble = new AsyncDrawable(
            getResources(), mPlaceHolderBitmap, task);
        imageView.setImageDrawable(asyncDrawable);
        task.execute(resId);
    }
}

在上面的代码示例中，cancelPotentialWork方法检查是否有另一个正在执行的任务与该ImageView关联了起来，如果已经关联，通过执行cancel()方法来取消另一个任务。在少数情况下，新创建的任务数据可能会与已经存在的任务相吻合，这样就不需要进行下一步动作了。下面是cancelPotentialWork方法的实现

public static boolean cancelPotentialWork(int data , ImageView imageView){
    final BitmapWorkerTask bitmapWorkerTask = getBitmapWorkerTask(imageView);
    if(bitmapWorkerTask != null){
        final int bitmapData = bitmapWorkerTask.data;
        if(bitmapData == 0 || bitmapData != data){
            // cancel previous task
            bitmapWorkerTask.cancel(true);
        }else{
            // the same work is already in progress
            return false;
        }
    }
    // No task associated with the ImageView , or an existing task was cancelled
    return true;
}

在上面的代码中有一个辅助方法：getBitmapWorkerTask(),它被用作检索AsyncTask是否已经被分配到指定的ImageView：

private static BitmapWorkerTask getBitmapWorkerTask(ImageView imageView){
    if(imageView != null){
        final Drawable drawable = imageView.getDrawable();
        if(drawable instanceof AsyncDrawable){
            final AsyncDrawable asyncDrawable = (AsyncDrawable) drawable;
            return asyncDrawable.getBitmapWorkerTask();
        }
    }
    return null;
}

最后一步是在BitmapWorkerTask的onPostExecute()方法中做更新操作：

class BitmapWorkerTask extends AsyncTask{
    ...
    @Override
    protected void onPostExecute(Bitmap bitmap){
        if(isCancelled()){
            bitmap = null;
        }
        if(imageViewReference != null && bitmap != null){
            final ImageView imageView = imageViewReference.get();
            final BitmapWorkerTask bitmapWorkerTask = 
                getBitmapWorkerTask(imageView);
            if(this == bitmapWorkerTask && imageView != null){
                imageView.setImageBitmap(bitmap);
            }
        }
    }
}

这个方法不仅适用于ListView和GridView控件，在那些需要循环利用子视图的控件中同样适用：只要设置图片到ImageView的地方调用loadBitmap方法。
